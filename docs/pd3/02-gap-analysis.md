# PD3 Gap Analysis — White-Paper Claims vs Repo Reality

> **Chosen scope level for PD3:** _[Minimum / Target / Stretch]_ — the team writes the choice here on day 1.

This document inventories every claim the long white paper makes about the system, marks each one against the actual state of the repository at the time of writing, and assigns a PD3 priority. The priority labels are:

- `MUST` — the demo cannot pass its verifiability test without this.
- `SHOULD` — visible in the demo and improves the verifiability story; build if Target scope.
- `STRETCH` — claimed in the white paper, not blocking for PD3; build only after MUST + SHOULD are green.
- `POST-PD3` — explicitly out of scope for the course delivery; documented as future work.

The PD3 reviewer can open the contract on Polygonscan and see whether it is behind a proxy.

## 6. White-paper claims that must be reconciled before submission

Whichever scope level the team picks, these specific sentences in the long white paper currently overshoot the implementation. Before submitting PD3, either ship the feature or soften the sentence. Pick one per row.

| Whitepaper claim (paraphrased) | Build it (Stretch) | Soften the sentence (Minimum / Target) |
| ------------------------------ | ------------------ | -------------------------------------- |
| "UUPS proxy with 14-day TimelockController" | Wire `UUPSUpgradeable` and `TimelockController` from OpenZeppelin; deploy through proxy | "The production deployment will wrap the contracts behind a UUPS proxy gated by a 14-day TimelockController. The PD3 demo uses direct deployments." |
| "3-of-5 Gnosis Safe (City Hall ×3, UFSC ×1, TCE-SC ×1)" | Create a Safe on Amoy, transfer admin to it | "The multisig is institutional and contingent on partner enrollment; PD3 demo uses a single EOA admin." |
| "ERC-4337 paymaster funded by the City Hall" | Integrate a paymaster (e.g., Stackup) | "Wallet abstraction is provided by a custodial demo wallet generated by the backend; ERC-4337 paymaster is on the production roadmap." |
| "Chainlink Automation calls `closeCycle()` at `closesAt`" | Register an upkeep on Amoy with LINK | "Demo closes the cycle by a manual transaction at `closesAt`; Chainlink Automation is on the production roadmap." |
| "audited Solidity contracts" (short paper) | Get an actual audit (impossible in PD3 timeframe) | "Tested under a 40-test Hardhat suite with CI lint+coverage; external audit contracted before any binding deployment." |

It is honest to leave the claims in the white paper as _production intent_ and explain that PD3 is a working demo that stops short of the full production configuration. The pitch video should make this explicit (one sentence is enough). The reviewer will look at Polygonscan; the contracts must match what the demo says they are.
